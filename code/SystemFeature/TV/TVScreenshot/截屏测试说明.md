# 截屏测试设计说明

## 1. 文档目的

- 说明截屏应用工程测试设计的总体思路
- 解释测试覆盖维度与设计点，证明测试充分性
- 为后续补测、回归和版本交付提供统一的测试依据

## 2. 测试对象

- `product/TV`：TV 入口模块，负责页面展示、服务能力接入和业务编排
- `features/screenshot_tv`：HAR 模块，负责截屏、图片处理、媒体库写入等核心能力
- `product/TV/src/ohosTest/ets/test`：自动化测试集，聚合功能与质量属性测试

## 3. 测试设计原则

- 主链路优先：先保证“触发截屏 -> 生成预览 -> 保存图片 -> 提示反馈”主流程稳定
- 核心对象拆分：将 `ServiceExtAbility`、`ViewModel`、`ScreenShotModel` 分层验证，避免问题定位困难
- 风险驱动设计：围绕系统权限、异步回调、资源占用、窗口管理等高风险点重点加测
- 质量属性补强：除功能外，补充性能、安全、兼容性、可访问性、国际化等非功能测试
- 回归可复用：将典型缺陷场景沉淀为稳定用例，便于版本演进时持续复测

## 4. 测试要素与设计点

| 测试维度 | 设计点 | 充分性说明 |
|---|---|---|
| 功能正确性 | 截屏触发、预览展示、保存图片、提示反馈 | 覆盖用户最核心业务路径，确保可用性 |
| 服务能力 | 系统动作接收、服务入口、生命周期 | 覆盖 TV 截屏由系统动作触发的关键入口 |
| 模型能力 | 截屏、图片编码、文件命名、媒体库写入 | 覆盖 HAR 核心逻辑，保证数据流闭环 |
| 视图编排 | 预览关闭、保存动作、结束流程 | 覆盖页面层与模型层之间的状态传递 |
| 集成场景 | 多步骤链路、跨模块调用、端到端状态一致性 | 防止单元测试无法识别的链路断裂问题 |
| 并发与竞态 | 高频触发、重复请求、互斥保护 | 防止多次截屏造成资源冲突或状态错乱 |
| 混沌与异常 | 服务中断、写入失败、只读文件系统、异常回调 | 验证容错与恢复能力，降低极端场景风险 |
| 数据完整性 | 截屏内容、保存状态、文件名唯一性 | 保证用户看到的结果与落盘数据一致 |
| 性能 | 截屏耗时、保存时延、资源占用 | 适配 TV 端资源有限场景，避免卡顿 |
| 安全 | 权限校验、越权访问、敏感能力边界 | 截屏属于敏感能力，必须验证最小权限原则 |
| 兼容性 | 分辨率、DPI、语言环境组合 | 保证不同设备和区域环境下界面及逻辑一致 |
| 国际化 | 多语言文案、RTL 布局、文件名格式 | 保证全球化场景可用 |
| 可访问性 | 焦点到达、无障碍识别、窗口可见性 | 保证大屏及辅助功能场景下的可操作性 |
| 回归 | 历史问题固化、基线场景复测 | 确保修复问题不会在后续版本中复发 |

## 6. 充分性说明

- 从纵向链路看，已覆盖系统触发、服务接入、模型处理、页面展示、结果保存的完整主流程
- 从横向质量看，已覆盖功能、性能、安全、兼容性、国际化、可访问性、异常恢复和回归八大类风险，其中兼容性已细分为总入口与 A/B/C 分类场景
- 从对象分层看，已对入口层、业务层、模型层、页面层分别设计用例，定位问题更直接
- 从场景复杂度看，既包含正常路径，也包含高频操作、异常故障、权限缺失、多设备差异等复杂场景
- 从维护角度看，测试目录已按维度拆分，适合后续继续增量扩展

## 7. 后续建议

- 新增需求时优先补充对应维度测试，再进入功能合并
- 每次修改 `ServiceExtAbility`、`ViewModel`、`ScreenShotModel` 后，至少执行对应专项测试与回归测试
- 如后续引入更多设备类型，建议继续扩展兼容性矩阵与性能基线
