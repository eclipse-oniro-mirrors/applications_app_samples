# 录屏测试说明

## 1. 文档目标

本文档用于说明 `TVScreenRecorder` 自动化测试的设计思路、覆盖维度、充分性依据与后续扩展方向。文档聚焦“为什么这样设计测试”和“当前测试为何具备较强覆盖能力”，用于辅助代码评审、测试审计和后续扩展。

## 2. 测试设计原则

### 2.1 分层设计

测试按以下层次拆分，避免所有用例堆叠在单一场景层：

- Foundation 层：提供断言、数据构造器、Mock、计数器、校验和与场景夹具
- Domain 层：围绕 `SRManager`、窗口管理、服务输入等核心域对象展开深度测试
- Scene 层：模拟权限流、Recent、状态栏、浮窗、提示页、焦点移动等真实使用场景
- Quality 层：补充性能、长稳压力、安全、契约、无障碍、布局边界等非功能维度

这一分层设计确保测试既能覆盖基础行为正确性，也能覆盖复杂交互与质量属性。

### 2.2 高复用设计

测试中大量复用以下设计资产：

- `FoundationAdvancedFixtures.ets`
- `TestAsserts.ets`
- `PerformanceSceneHarness.ets`
- `ContractSecurityHarness.ets`
- `AccessibilityLayoutHarness.ets`

通过复用夹具、断言和 Harness，避免重复造轮子，使测试更容易扩展到更深的参数矩阵和更长的回归链路。

### 2.3 矩阵化设计

关键场景不只测试单点，而是采用组合矩阵扩展覆盖：

- 分辨率 × 旋转角度
- 画像配置 × 灰度桶
- Recent × 状态栏 × 冷启动恢复
- 生命周期状态 × 故障模式
- 输入方向 × 焦点位置 × 页面渲染

矩阵化设计的意义在于降低“单点通过、组合失败”的漏测风险。

## 3. 测试对象拆解

### 3.1 录屏主流程

主流程测试聚焦以下对象：

- 录屏服务扩展能力
- `SRManager` 录屏状态机
- 悬浮计时窗
- 录屏结束提示页
- 分辨率不支持提示页
- 输入按键消费链路
- Recent / 状态栏 / 存储投影链路

### 3.2 关键风险点

围绕 TV 录屏场景，重点识别如下风险：

- 录屏启动与停止流程出现非法状态切换
- 悬浮窗与提示页并发创建导致状态污染
- 快捷键结束录屏时状态未同步回主流程
- 异常恢复后 Recent、状态栏和存储域残留脏数据
- 分辨率、旋转、设备画像变化导致布局或功能退化
- 危险输入、路径穿越和不合法配置导致安全问题
- 无障碍树缺失、文本不可读或焦点无法正常导航

这些风险点直接决定了当前测试设计的重点方向。

## 4. 覆盖维度设计

### 4.1 功能正确性

功能测试不只验证“能否录屏”，而是拆成可验证的细粒度行为：

- 创建录屏任务
- 启动录屏
- 停止录屏
- 更新状态
- 写入结果
- 弹出提示窗
- 关闭提示页
- Recent 列表更新
- 状态栏映射更新

这样可以定位失败点，而不是只得到“主流程失败”的粗粒度结论。

### 4.2 异常流与恢复

异常流设计覆盖：

- 启动失败
- 停止失败
- 缺失依赖键
- 不支持分辨率
- 危险输入与不合法参数
- 状态恢复失败前后的数据比对

恢复类测试要求不仅校验接口返回值，还要求校验：

- 当前状态是否回到 `idle`
- Recent 是否清理
- 状态栏是否回归一致
- 持久化域是否存在污染

这保证了异常测试不仅关注“报错”，也关注“报错之后系统是否恢复正确”。

### 4.3 并发与长稳

TV 形态下，用户可能连续执行开始、停止、切换焦点、打开弹窗等动作，因此需要覆盖：

- 并发窗口创建
- 高频输入组合
- 长轮次混合操作
- 性能指纹双遍比对
- 计数器与校验和稳定性校验

长稳测试采用混合动作回放，而不是单一动作重复，以更贴近真实使用路径。

### 4.4 性能与兼容性

性能设计不只关注单个耗时值，而是通过以下方式提升充分性：

- 记录 start / stop / focus 多类时延
- 使用平均值、分位值和最大值等指标观察趋势
- 对同一回放序列做双遍指纹比对，验证稳定性
- 在不同分辨率、旋转角度、配置画像下重复执行

兼容性设计强调多维组合，避免只在默认环境通过。

### 4.5 安全与契约

安全与契约测试设计包含：

- Schema 键存在性检查
- 状态快照一致性比对
- 灰度键、发布桶与画像之间的映射
- 危险输入过滤
- 路径穿越阻断
- 持久化命名空间隔离

这一部分用于降低未来版本演进过程中的接口漂移和数据污染风险。

### 4.6 可达性与易用性

针对 TV 端操作特点，测试设计特别覆盖：

- 无障碍树是否存在根节点和有效父子关系
- 关键文案是否可读
- 焦点是否能按遥控方向移动
- 布局在不同分辨率和旋转角度下是否越界

这部分是判断 TV 大屏可用性的重要补充。

## 5. 充分性说明

### 5.1 为什么认为当前测试具备较强充分性

当前测试充分性主要来自以下几个方面：

- 覆盖层次完整：从断言夹具层到场景回归层都有对应测试
- 覆盖维度：功能、异常、性能、安全、契约、无障碍、布局均已纳入
- 覆盖方式多样：单点验证、矩阵组合、长稳回放、双遍指纹、状态快照并存
- 数据校验细化：不仅校验返回码，还校验状态、Recent、状态栏、存储与快照
- 测试资产可复用：便于继续扩展而不破坏既有结构

## 6. 当前测试目录与职责

| 目录 | 职责 |
|---|---|
| foundation | 断言、Mock、夹具、基础构造 |
| srmanager | 核心录屏管理逻辑测试 |
| ability | 生命周期和编排测试 |
| window | 窗口与浮窗测试 |
| ui | 界面、Recent、无障碍、布局测试 |
| service | 输入与命令流测试 |
| performance | 性能、极限与长稳测试 |
| contract | 契约、安全与持久化测试 |


## 7. 结论

`TVScreenRecorder` 当前测试设计已经形成“基础设施 + 核心域 + 真实场景 + 非功能质量”的组合式覆盖结构。该结构具备良好的扩展性、可维护性和风险发现能力，可以作为本工程后续持续集成、版本回归和测试审计的核心基础。
